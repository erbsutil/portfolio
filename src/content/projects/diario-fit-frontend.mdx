teamSize: 1
outcomeSummary: "Dashboard de saúde e nutrição acessível via link mágico gerado pelo Telegram, sem nenhum login tradicional — o utilizador abre o painel no telemóvel e vê os seus dados de hoje, heatmaps de 90 dias e ranking do grupo, tudo em tempo real a partir do Supabase."
overview: "O Diário Fit é uma plataforma de acompanhamento nutricional e de treino cujo ponto de entrada é um bot Telegram. O utilizador regista refeições e treinos conversacionalmente e, quando quer uma visão agregada, pede 'painel' ao bot que lhe devolve um URL com token. Este repositório — Next.js (App Router) — é o dashboard web que consome esse link: carrega dados do Supabase via service role no servidor, monta gráficos de consistência (heatmaps), totais do dia, volume de musculação e ranking de grupo, e entrega tudo sem pedir password ao utilizador."
problem: "O bot Telegram já processava e armazenava entradas nutricionais e de treino, mas não existia forma de visualizar tendências ao longo do tempo: o utilizador vivia 'às cegas' quanto à consistência semanal, progresso de volume e comparativo com parceiros de grupo. O desafio era criar esse painel sem introduzir fricção de autenticação — forçar login anularia a fluidez do ecossistema conversacional."
constraints:
  - "Sem fluxo de login — o acesso é 100% via token UUID na URL, gerado pelo bot."
  - "RLS do Supabase ativado: a chave anon não tem permissão para ler user_profiles; o servidor precisa de service role."
  - "Privacidade de grupo: outros membros do mesmo grupo só podem expor colunas mínimas (telegram_id, created_at, calories) — raw_input, parsed_data e exercises jamais chegam ao browser de terceiros."
  - "Fuso horário per-user: o conceito de 'hoje' é o dia civil no fuso do perfil (user_profiles.timezone), não no fuso do servidor nem do browser."
  - "Hidratação SSR↔CSR: componentes que dependem do relógio (heatmaps, 'Hoje/Ontem') têm de ser renderizados no cliente ou usar suppressHydrationWarning para evitar mismatch."
  - "Performance mobile: o link é enviado pelo Telegram e aberto num telemóvel; o bundle e as imagens precisam de ser enxutos."
approach: "A arquitetura segue o modelo Server Component → Client Component do Next.js App Router. O Server Component da rota /status/[token] chama getDashboardData() que (1) valida e sanitiza o token, (2) lê o perfil com service role bypassando RLS, (3) busca entradas do titular com select completo e entradas de peers com select mínimo em paralelo via Promise.all, (4) calcula os totais de hoje usando localDay.ts para derivar o dia civil correto, e (5) remove os campos sensíveis antes de serializar. Os Client Components recebem os dados já prontos via props e tratam apenas de interação, animação e layout responsivo."
keyDecisions:
  - decision: "Link mágico com service role no servidor, em vez de auth tradicional"
    reasoning: "O utilizador já está autenticado no Telegram. Criar um segundo fluxo de login (email/password ou OAuth) adicionaria fricção que contradiz o modelo conversacional. O token UUID de curta duração (~7 dias) gerado pelo bot funciona como credencial delegada: o servidor valida, usa service role para ler o perfil e nunca expõe a service key ao browser."
    alternatives:
      - "Supabase Auth com magic link por e-mail (descartado: utilizadores não forneceram e-mail)"
      - "Chave anon com RLS permissiva (descartado: expõe dados de outros utilizadores)"
  - decision: "Select diferenciado por papel (owner vs peer)"
    reasoning: "Para o gráfico de consistência do grupo é necessário saber se cada membro comeu naquele dia e quantas calorias — mas revelar o raw_input ou parsed_data ao browser de outra pessoa viola privacidade. A solução foi fazer dois SELECTs paralelos: select('*') para o titular e select('telegram_id,created_at,calories') para os peers. O TypeScript enforce isso com tipos distintos (DashboardOwnerDietEntry vs DashboardPeerDietEntry), impedindo acesso acidental a campos protegidos."
    alternatives:
      - "RLS view que filtra colunas (descartado: mais complexidade no Supabase, menos visibilidade no código)"
  - decision: "localDay.ts — cálculo de dia civil via Intl.DateTimeFormat, sem biblioteca de datas"
    reasoning: "O bot Telegram (Windmill/Bun) já usa a mesma lógica de busca binária para encontrar o início do dia local em UTC. Replicar exatamente o mesmo ficheiro no front-end garante que 'hoje' no dashboard coincide com 'hoje' no bot — crucial para que os totais do painel batam com o que o utilizador registou na conversa. Usar uma biblioteca diferente (date-fns-tz, Luxon) introduziria risco de divergência em fusos com DST."
    alternatives:
      - "date-fns-tz (descartado: risco de divergência de DST face ao Windmill)"
      - "Moment Timezone (descartado: bundle pesado, comportamento diferente em edge runtimes)"
  - decision: "Heatmap estilo GitHub Contributions (7 linhas × N semanas)"
    reasoning: "O utilizador já está familiarizado com o padrão de contribuições do GitHub. Orientar as colunas por semana (mais antigo à esquerda) e as linhas por dia da semana (Seg–Dom) permite identificar padrões de comportamento semanal de relance — ex: 'nunca me exercito às sextas'. A alternativa de uma lista linear de dias ocultaria esses padrões."
    alternatives:
      - "Gráfico de barras diário (descartado: não revela padrões semanais)"
      - "Calendário mensal estilo Google Calendar (descartado: 90 dias não cabem sem scroll)"
  - decision: "Volume de treino filtra apenas musculação (6 grupos de força)"
    reasoning: "O bot regista cardio com um padrão simbólico de 0 kg × 1 rep. Se esse padrão entrasse no cálculo de kg×reps, todos os dias de corrida apareceriam com volume zero e distorceriam o gráfico. A decisão foi filtrar pelos 6 grupos de força (Peito, Costas, Pernas, Ombros, Braços, Core) e excluir qualquer série com 0 kg × 1 rep."
    alternatives:
      - "Mostrar cardio como métrica separada (backlog, aguarda suporte de distância/tempo no bot)"
  - decision: "server-only para supabaseAdmin.ts"
    reasoning: "O import 'server-only' do Next.js gera um erro de build se o módulo for acidentalmente importado num Client Component. Como supabaseAdmin.ts contém a SUPABASE_SERVICE_ROLE_KEY, expô-la ao bundle do browser seria uma vulnerabilidade crítica. O guard em tempo de compilação elimina essa classe de erro sem depender de revisão de código."
    alternatives:
      - "Variável de ambiente sem prefixo NEXT_PUBLIC_ (proteção suficiente mas silenciosa)"

  qualitative: "Utilizadores passaram a ter visibilidade de tendências que antes eram invisíveis no chat: consistência semanal de refeições e treinos em 90 dias, padrões de volume de musculação ao longo do tempo e comparativo motivacional com o grupo — tudo acessível com um tap no link do Telegram, sem instalar nada nem criar conta."
learnings:
  - "Manter lógica de domínio crítica (dia civil por fuso) como um ficheiro partilhado conceitualmente entre backend e frontend — mesmo que não seja literalmente o mesmo módulo importado — elimina uma categoria inteira de bugs de sincronização. A decisão de alinhar localDay.ts do front com o localDay.ts do Windmill/Bun foi mais valiosa do que qualquer biblioteca."
  - "O modelo de tipos distintos por papel (owner vs peer) no TypeScript não é apenas segurança de privacidade: é documentação viva de quais campos chegam ao browser de quem. Qualquer PR que quebre essa distinção falha em compilação."
  - "Server Components do App Router simplificam drasticamente o fluxo de dados sensíveis: a service key nunca precisa de sair do servidor, e não há endpoint de API a proteger — o componente de rota é o próprio guardião."
  - "Heatmaps com interação (sheet/tooltip por célula) têm um custo de hidratação SSR↔CSR não óbvio. A solução com suppressHydrationWarning nos pontos certos e renderização client-only nos componentes que dependem do relógio local é mais simples do que tentar sincronizar servidor e browser no mesmo instante de 'agora'."
  - "Filtrar dados na query (select mínimo para peers) é mais seguro e performático do que filtrar em memória no servidor — e é a única forma de garantir que dados sensíveis nunca saem da base de dados, mesmo que haja um bug de lógica downstream."
