title: "Diário Fit — Bot Telegram & Dashboard Web"
role: "Fullstack Engineer & Product Owner"
year: 2025
duration: "Ongoing"
teamSize: 1
outcomeSummary: "Bot de diário alimentar e de treino em produção no Telegram com análise multimodal via Gemini, hospedado num servidor local próprio com Docker a custo zero, e dashboard web acessível via link mágico — sem nenhum login adicional."
overview: "O Diário Fit é uma plataforma de acompanhamento nutricional e de treino construída para uso familiar. O backend vive inteiramente no Windmill — plataforma de workflows com runtime Bun — auto-hospedado num servidor local via Docker a custo zero, e expõe o produto via bot no Telegram. Quando o utilizador quer uma visão agregada, pede 'painel' ao bot, que devolve um URL com token; esse link abre um dashboard Next.js que carrega dados do Supabase, exibe heatmaps de consistência de 90 dias, totais do dia, volume de musculação e ranking de grupo — tudo sem pedir password. Meu papel foi de engenheiro e dono do produto: projetei a arquitetura completa, escrevi todos os scripts e componentes, defini o protocolo de IA e mantive o sistema em evolução contínua (versão 11.0.0 em abril/2026)."
problem: "A família precisava de uma forma de registrar refeições e treinos no dia a dia sem fricção — sem formulários, sem apps extras para instalar. O canal já existente era o Telegram. O desafio técnico era transformar mensagens livres em texto, foto e voz em dados estruturados e persistidos de forma confiável, e depois oferecer uma visualização de tendências sem introduzir um segundo fluxo de autenticação — forçar login anularia a fluidez do ecossistema conversacional."
constraints:
  - "Infraestrutura self-hosted a custo zero: Windmill rodando num servidor local próprio via Docker — sem custo de cloud, mas com responsabilidade de manter o daemon e a conectividade do webhook."
  - "Runtime Bun no worker do Windmill não suporta imports relativos entre scripts, obrigando bundle único por esbuild."
  - "API do Telegram encerra conexão webhook após ~60 s; jobs com Gemini + ElevenLabs se aproximam desse limite."
  - "Custo zero de GPU: análise de imagem, transcrição de voz e geração de texto devem reutilizar a mesma chave Gemini."
  - "Grupo de Telegram: bot precisa ignorar ruído de conversa sem ser invasivo, mas capturar mensagens de dieta/treino sem falsos negativos."
  - "Idempotência obrigatória: webhook pode entregar o mesmo update mais de uma vez em retentativas."
  - "Sem fluxo de login no dashboard — o acesso é 100% via token UUID na URL, gerado pelo bot."
  - "RLS do Supabase ativado: a chave anon não tem permissão para ler user_profiles; o servidor Next.js precisa de service role."
  - "Privacidade de grupo: outros membros só podem expor colunas mínimas (telegram_id, created_at, calories) — raw_input, parsed_data e exercises jamais chegam ao browser de terceiros."
  - "Fuso horário per-user: o conceito de 'hoje' é o dia civil no fuso do perfil (user_profiles.timezone), não no fuso do servidor nem do browser."
  - "Hidratação SSR↔CSR: componentes que dependem do relógio (heatmaps, 'Hoje/Ontem') têm de ser renderizados no cliente ou usar suppressHydrationWarning para evitar mismatch."
approach: "O backend roda no Windmill self-hosted num servidor local via Docker — sem custo de cloud e com controlo total sobre o ambiente. O ponto de entrada é um webhook do Telegram que dispara um job Bun. Toda a lógica do bot vive em módulos TypeScript em src/, compilados num bundle único via esbuild antes de cada deploy. O Gemini 2.5 Flash centraliza análise de texto, interpretação de foto e transcrição de voz. O Supabase (PostgreSQL) armazena perfis, entradas diárias e treinos. O dashboard segue o modelo Server Component → Client Component do Next.js App Router: o Server Component valida o token, lê o perfil com service role bypassando RLS, busca entradas do titular e de peers em paralelo via Promise.all, calcula os totais do dia usando localDay.ts e remove campos sensíveis antes de serializar. A lógica de 'dia civil' (timezone-aware) foi extraída em localDay.ts, espelhado entre backend e frontend para garantir que 'hoje' no bot e no dashboard seja sempre o mesmo intervalo UTC."
keyDecisions:
  - decision: "Windmill self-hosted via Docker em servidor local, em vez de cloud gerenciada"
    reasoning: "Roda num servidor local próprio com Docker, eliminando custo de cloud sem abrir mão de logs estruturados por execução, retry automático e agendamento nativo. O tradeoff é a responsabilidade de manter o daemon e garantir a conectividade do webhook — algo aceitável para um projeto familiar com acesso físico à máquina."
    alternatives:
      - "Windmill Cloud (descartado: custo mensal desnecessário para o volume familiar)"
      - "Railway ou Fly.io com servidor Express/Hono (descartado: mais complexidade operacional sem os benefícios do Windmill)"
      - "Cloudflare Workers (descartado pelo limite de runtime e ausência de Bun nativo)"
  - decision: "Bundle único por esbuild em vez de múltiplos scripts com imports relativos"
    reasoning: "O loader Bun no worker do Windmill falha ao resolver imports relativos entre arquivos. Bundlar tudo num único arquivo main.ts entregável resolve o problema sem alterar a estrutura de desenvolvimento, que mantém módulos separados em src/."
    alternatives:
      - "Inlining manual do código (descartado: inviável de manter)"
      - "Usar Deno runtime (descartado: exigiria reescrever dependências e afastaria do ecossistema npm)"
  - decision: "Gemini como modelo único para texto, visão e transcrição de voz"
    reasoning: "Reduz o número de chaves de API, simplifica o rate limiting e mantém o custo operacional próximo de zero para o volume familiar. A transcrição via Gemini tem qualidade equivalente ao Whisper para PT-BR no contexto de alimentação e treino."
    alternatives:
      - "OpenAI Whisper para transcrição (mantido como opcional via OPENAI_API_KEY, usado se disponível)"
      - "Modelo local de visão (inviável sem GPU própria)"
  - decision: "Gate de grupo com heurísticas de score em vez de comandos explícitos"
    reasoning: "Num supergrupo familiar, exigir / ou menção para cada mensagem criaria fricção inaceitável. O gate pondera palavras de dieta, verbos de treino, modalidades (zumba, natação, maratona…), duração por extenso e padrões de vocativo. Isso permite que o bot 'ouça' organicamente sem invadir conversas irrelevantes."
    alternatives:
      - "Modo só-comando (descartado: os utilizadores não querem digitar prefixos para cada refeição)"
      - "IA para decidir relevância (descartado: custo e latência para mensagens que devem ser ignoradas silenciosamente)"
  - decision: "Link mágico com service role no servidor, em vez de auth tradicional para o dashboard"
    reasoning: "O utilizador já está autenticado no Telegram. O token UUID de curta duração (~7 dias) gerado pelo bot funciona como credencial delegada: o servidor valida, usa service role para ler o perfil e nunca expõe a service key ao browser."
    alternatives:
      - "Supabase Auth com magic link por e-mail (descartado: utilizadores não forneceram e-mail)"
      - "Chave anon com RLS permissiva (descartado: expõe dados de outros utilizadores)"
  - decision: "Select diferenciado por papel (owner vs peer)"
    reasoning: "Para o gráfico de consistência do grupo é necessário saber se cada membro comeu naquele dia — mas revelar o raw_input ou parsed_data ao browser de outra pessoa viola privacidade. A solução foi dois SELECTs paralelos: select('*') para o titular e select('telegram_id,created_at,calories') para os peers. O TypeScript enforce isso com tipos distintos, impedindo acesso acidental a campos protegidos."
    alternatives:
      - "RLS view que filtra colunas (descartado: mais complexidade no Supabase, menos visibilidade no código)"
  - decision: "localDay.ts espelhado manualmente entre backend e frontend"
    reasoning: "A lógica de janela UTC do dia civil precisa ser idêntica nos dois projetos para que as queries do bot e do dashboard retornem os mesmos registos. Criar um pacote npm compartilhado adicionaria complexidade de publicação e versionamento desnecessária para um time de uma pessoa. O espelho manual com teste automatizado (test_local_day.mjs) garante sincronização sem overhead."
    alternatives:
      - "Pacote npm privado compartilhado (descartado: overhead de publish/versioning)"
      - "Duplicação sem teste (descartado: divergência silenciosa entre projetos)"
  - decision: "Idempotência via update_id gravado no banco antes de chamar a IA"
    reasoning: "O Telegram faz retentativas se o webhook não responder a tempo. Sem idempotência, a mesma refeição poderia ser gravada duas vezes. Checar o update_id antes de qualquer chamada ao Gemini é a barreira mínima que resolve o problema com uma query simples."
    alternatives:
      - "Redis/cache externo para deduplicação (descartado: dependência extra sem ganho real no volume atual)"
  - decision: "server-only para supabaseAdmin.ts no dashboard"
    reasoning: "O import 'server-only' do Next.js gera erro de build se o módulo for acidentalmente importado num Client Component. Como supabaseAdmin.ts contém a SUPABASE_SERVICE_ROLE_KEY, expô-la ao bundle do browser seria uma vulnerabilidade crítica. O guard em tempo de compilação elimina essa classe de erro sem depender de revisão de código."
    alternatives:
      - "Variável de ambiente sem prefixo NEXT_PUBLIC_ (proteção suficiente mas silenciosa)"
  - decision: "Heatmap estilo GitHub Contributions (7 linhas × N semanas)"
    reasoning: "O utilizador já está familiarizado com o padrão de contribuições do GitHub. Orientar as colunas por semana e as linhas por dia da semana (Seg–Dom) permite identificar padrões de comportamento semanal de relance — ex: 'nunca me exercito às sextas'. Uma lista linear de dias ocultaria esses padrões."
    alternatives:
      - "Gráfico de barras diário (descartado: não revela padrões semanais)"
      - "Calendário mensal estilo Google Calendar (descartado: 90 dias não cabem sem scroll)"

learnings:
  - "Bundling como solução de compatibilidade: quando o ambiente de execução impõe restrições ao sistema de módulos, uma etapa de build resolve o problema sem comprometer a organização do código-fonte."
  - "Gate de relevância em grupo é um produto, não só código: as heurísticas de filtragem de mensagens exigiram mais iterações do que qualquer outra parte técnica, porque o critério de 'relevante' é essencialmente uma decisão de UX."
  - "Espelhar lógica crítica (dia civil por fuso) com teste automatizado é mais seguro do que criar abstração prematura: o localDay.ts duplicado com suite de testes provou ser mais confiável do que um pacote compartilhado sem processo de publicação definido, e garante que bot e dashboard calculam 'hoje' de forma idêntica."
  - "Idempotência barata vale mais do que idempotência sofisticada: checar um campo no banco antes de chamar a IA resolve 99% dos casos de retentativa do Telegram sem adicionar dependências."
  - "O modelo de tipos distintos por papel (owner vs peer) no TypeScript não é apenas segurança de privacidade: é documentação viva de quais campos chegam ao browser de quem. Qualquer PR que quebre essa distinção falha em compilação."
  - "Server Components do App Router simplificam o fluxo de dados sensíveis: a service key nunca precisa sair do servidor, e não há endpoint de API a proteger — o componente de rota é o próprio guardião."
